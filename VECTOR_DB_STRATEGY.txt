2. QUE TENES QUE HACER AHORA (MANUAL)
========================================================================

PASO INMEDIATO: Correr la migracion SQL en Supabase.

  1. Ir a Supabase Dashboard → SQL Editor → New query
  2. Pegar y ejecutar este SQL:

     CREATE EXTENSION IF NOT EXISTS vector;

     CREATE TABLE dish_embeddings (
       id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
       user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
       dish_name TEXT NOT NULL,
       ingredients JSONB NOT NULL,
       total_kcal INTEGER NOT NULL,
       prep_minutes INTEGER,
       human_portion TEXT,
       embedding vector(768) NOT NULL,
       meal_type TEXT CHECK (meal_type IN ('desayuno','almuerzo','cena','snack')),
       date_planned DATE,
       created_at TIMESTAMPTZ DEFAULT NOW()
     );

     CREATE INDEX idx_dish_embed_user ON dish_embeddings(user_id);
     CREATE INDEX idx_dish_embed_vector ON dish_embeddings
       USING hnsw (embedding vector_cosine_ops);

     CREATE OR REPLACE FUNCTION find_similar_dishes(
       query_embedding vector(768),
       user_uuid UUID,
       match_threshold FLOAT DEFAULT 0.7,
       match_count INT DEFAULT 10
     )
     RETURNS TABLE (
       id UUID, dish_name TEXT, ingredients JSONB, total_kcal INTEGER,
       prep_minutes INTEGER, human_portion TEXT, meal_type TEXT, similarity FLOAT
     )
     LANGUAGE sql STABLE
     AS $$
       SELECT d.id, d.dish_name, d.ingredients, d.total_kcal,
              d.prep_minutes, d.human_portion, d.meal_type,
              1 - (d.embedding <=> query_embedding) AS similarity
       FROM dish_embeddings d
       WHERE d.user_id = user_uuid
         AND 1 - (d.embedding <=> query_embedding) > match_threshold
       ORDER BY d.embedding <=> query_embedding
       LIMIT match_count;
     $$;

  3. Verificar que funciona:
     SELECT * FROM dish_embeddings LIMIT 1;
     (debe devolver 0 rows, sin error)

  4. Usar la app normalmente — pedir un plan semanal, aplicarlo.
     Despues verificar en Supabase:
     SELECT dish_name, meal_type, created_at
     FROM dish_embeddings ORDER BY created_at DESC LIMIT 10;

Una vez que la tabla existe, el pipeline ya esta activo. Cada vez que
la IA genere una comida y el usuario la aplique, el embedding se genera
y guarda automaticamente en background.

3. COMO FUNCIONA EL PIPELINE ACTUAL
========================================================================

El flujo es 100% async y fire-and-forget:

  Usuario pide comida → Gemini genera → Frontend recibe respuesta
    ├─ Aplica al calendario (existente, no cambia)
    └─ POST /api/ai/embed (async, no bloquea)
        ├─ Genera embedding con text-embedding-004 (768 dimensiones)
        ├─ Chequea dedup (nombre exacto por usuario)
        └─ INSERT en dish_embeddings

Si el embedding falla, la comida ya esta en el calendario. El usuario
nunca se entera. Solo hay un console.warn en el browser.

ENDPOINT DE BUSQUEDA (listo pero sin usar en UI):
  POST /api/ai/search
  Body: { query: "algo con pollo", limit: 10, mealType: "almuerzo" }
  → Genera embedding del query
  → Llama find_similar_dishes() via RPC
  → Devuelve platos similares con score de similaridad

4. ESTRATEGIA PARA USAR LA DB VECTORIAL EN LA IA
========================================================================

La idea central: la DB vectorial le da MEMORIA a Gemini. Hoy Gemini
genera comidas "de cero" cada vez, sin saber que le gusto al usuario
antes. Con embeddings podemos inyectar contexto personalizado.

FASE 1: CONTEXTO HISTORICO (Module 3 — Historial)
-------------------------------------------------
Cuando el usuario pide "armame la semana", ANTES de llamar a Gemini:

  1. Buscar los ultimos 20-30 platos del usuario en dish_embeddings
  2. Agrupar por frecuencia (cuantas veces comio cada cosa)
  3. Inyectar en el system prompt de Gemini:

     HISTORIAL DE COMIDAS DE {NOMBRE}:
     - Pollo al horno con papas (8 veces, ultima: hace 2 dias)
     - Milanesas con ensalada (6 veces, ultima: hace 5 dias)
     - Tarta de espinaca (3 veces, ultima: hace 1 semana)
     Preferencias detectadas: le gustan los platos con pollo,
     cocina seguido, prefiere comidas rapidas.

  Resultado: Gemini genera planes que SABEN que le gusta al usuario.

FASE 2: DETECCION DE VARIEDAD
-------------------------------------------------
Antes de generar un plan semanal:

  1. Buscar platos de las ultimas 2 semanas
  2. Si hay platos muy repetidos (>3 veces), decirle a Gemini:

     VARIEDAD: {NOMBRE} comio "Pollo al horno" 5 veces en 2 semanas.
     Sugeri alternativas similares pero diferentes.

  3. Usar semantic search para encontrar platos SIMILARES pero no iguales:
     - Query: embedding del plato repetido
     - Threshold: 0.6-0.8 (similar pero no identico)
     - Sugerir esos como alternativas

  Resultado: planes mas variados, sin repetir lo mismo toda la semana.

FASE 3: SUSTITUCIONES INTELIGENTES
-------------------------------------------------
Cuando el usuario dice "cambiame el almuerzo":

  1. Tomar el plato actual → buscar su embedding
  2. Buscar platos similares del usuario (misma categoria calorica)
  3. Filtrar: que no sea el mismo, que respete restricciones
  4. Inyectar las opciones en el prompt de Gemini:

     El usuario quiere cambiar "Milanesas con ensalada".
     Platos similares que ya le gustaron:
     - Suprema al horno con pure (similarity: 0.85)
     - Pollo grillado con verduras (similarity: 0.78)
     Sugeri algo basado en estas preferencias o algo nuevo.

  Resultado: swaps que se sienten personalizados, no aleatorios.

FASE 4: BUSQUEDA SEMANTICA EN HISTORIAL
-------------------------------------------------
Para el tab de Historial (Module 3):

  1. Barra de busqueda: usuario escribe "algo con verduras"
  2. POST /api/ai/search con ese query
  3. Devuelve platos del historial que matchean semanticamente
  4. No es string matching — "algo con verduras" encuentra
     "Tarta de espinaca", "Wok de vegetales", "Ensalada cesar"

  Resultado: busqueda intuitiva, no necesita recordar nombres exactos.

FASE 5: RECOMENDACIONES PROACTIVAS (futuro)
-------------------------------------------------
Sin que el usuario pregunte:

  1. Analizar patrones: que come los lunes? que come cuando hace frio?
  2. Detectar "huecos": si nunca come pescado, sugerir incorporarlo
  3. Notificacion sutil: "Hace 2 semanas que no comes pescado.
     Esta semana podriamos incluir un salmon?"

  Esto es mas avanzado y depende de tener suficiente historial.

5. PERSONALIZACIONES PRIORITARIAS SEGUN EL CONTEXTO DEL PROYECTO
========================================================================

Dado que NutriKal es para familias argentinas (10-20 personas) y la
filosofia es "cero esfuerzo, cero ansiedad", las personalizaciones
mas valiosas son:

A) APRENDER GUSTOS SIN PREGUNTAR
   Problema: Hoy Gemini pregunta preferencias cada vez.
   Solucion: Con el historial de embeddings, Gemini ya sabe.
   Impacto: Menos preguntas → plan mas rapido → mejor UX.

B) VARIEDAD AUTOMATICA
   Problema: Sin memoria, Gemini puede sugerir lo mismo varias semanas.
   Solucion: Inyectar historial reciente → "no repitas estos platos".
   Impacto: Planes frescos cada semana sin que el usuario lo pida.

C) CONTEXTO ARGENTINO REAL
   Problema: Gemini a veces sugiere ingredientes no comunes en Argentina.
   Solucion: Los embeddings son de platos que el usuario REALMENTE comio.
   Al sugerir basado en historial, los platos son naturalmente argentinos.
   Impacto: Sugerencias mas realistas y practicas.

D) SWAP PERSONALIZADO
   Problema: "Cambiame el almuerzo" genera algo aleatorio.
   Solucion: Buscar en embeddings → sugerir algo similar a lo que le gusto.
   Impacto: Swaps que el usuario realmente va a aceptar.

E) DETECCION DE ABURRIMIENTO
   Problema: Usuario come lo mismo 3 semanas → se aburre → deja la app.
   Solucion: Detectar repeticion via embeddings → sugerir variantes.
   Impacto: Retencion a largo plazo.

F) SHOPPING OPTIMIZADO (futuro)
   Problema: Lista de compras cambia mucho semana a semana.
   Solucion: Si detectamos que compra siempre pollo, papas, huevos →
   sugerir planes que reutilicen ingredientes comunes.
   Impacto: Menos desperdicio, compras mas eficientes.
